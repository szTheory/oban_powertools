# Architecture Research: Worker Lifecycle & Safety

**Milestone:** v1.7 Worker Lifecycle & Safety
**Researched:** 2026-05-30
**Confidence:** HIGH — grounded in direct inspection of all named modules; no speculative gaps

---

## Integration Points

### 1. Worker Hooks (on_start / on_success / on_failure / on_discard)

**Existing seam:** `ObanPowertools.Worker.__using__/1` generates `perform/1` which calls `process/1`. The Oban executor already emits `[:oban, :job, :start]`, `[:oban, :job, :stop]`, and `[:oban, :job, :exception]` telemetry events at the right lifecycle moments (see `Oban.Queue.Executor`).

**Integration point:** The `perform/1` generated by `ObanPowertools.Worker` is the exact right place to call hooks. It is already the one function that wraps `process/1` with validation. The hooks slot in before `process/1` (`on_start`), and in a `try/rescue/catch` wrapper around `process/1` (`on_success`, `on_failure`, `on_discard`). No Oban internals need modification.

**Oban telemetry as source of truth:** Oban's executor already classifies the final outcome (`:success`, `:failure`, `:exhausted`, `:discard`, `:cancelled`). The Powertools hooks operate above the executor — they run inside `perform/1`, not as telemetry handlers. This is intentional: hooks should run synchronously before Oban commits the outcome to the database.

**Crash-caught contract:** Hooks must never crash the job. If a hook raises, `perform/1` catches it, logs a warning, and proceeds normally. The `rescue/catch` wrapping is a hard contract, not opt-in.

**Observe-only contract:** Hooks receive the `Oban.Job.t()` (post-cast to Args struct) and the outcome (for `on_success`, `on_failure`, `on_discard` — the return value from `process/1`). They cannot affect the job outcome. Return values are discarded.

### 2. Soft deadline: / timeout: Pass-through

**Existing seam:** `ObanPowertools.Worker.__using__/1` already passes `oban_opts` (everything except `:args` and `:limits`) directly to `use Oban.Worker`. This means `max_attempts:`, `queue:`, `tags:`, `unique:` all pass through today.

**Integration point:** `deadline:` and `timeout:` need different handling:

- `timeout:` — Oban already supports a `timeout/1` callback on the worker (`def timeout(_job), do: :timer.seconds(30)`). The Powertools macro can generate a `timeout/1` override when `timeout:` is declared in `use ObanPowertools.Worker, timeout: 30_000`. This is purely compile-time generation, no new data flow.

- `deadline:` — Oban has no native `deadline:` concept for jobs. `deadline:` in this context means "do not attempt the job if it has been waiting longer than N ms/seconds since `scheduled_at`". The implementation is: override `perform/1` to check `DateTime.diff(DateTime.utc_now(), job.scheduled_at)` before calling `process/1`, and return `{:discard, :deadline_exceeded}` if past the deadline. The deadline value can be stored as a module attribute (compile-time constant) or read from `job.meta["oban_powertools"]["deadline_ms"]` for dynamic use.

**No new opts stripping needed:** `deadline:` is a Powertools-only key and must be stripped before the `use Oban.Worker` call, analogous to how `:args` and `:limits` are stripped today.

### 3. Output Recording (generalising Workflow.Result)

**Existing seam:** `Workflow.Result` (`oban_powertools_workflow_results` table) is bound to `workflow_id` + `step_id` via non-nullable foreign keys. It exists only for workflow steps.

**Integration point:** A generalised `JobRecord` (working name) stores output for any Oban job, indexed by `oban_job_id` (integer FK to `oban_jobs`). The schema is nearly identical to `Workflow.Result` — `status`, `payload`, `payload_bytes`, `retention`, `redacted`, `summary`, `recorded_at`, `expires_at` — but the workflow-specific FK columns are replaced with `oban_job_id`.

Two design options for the existing `Workflow.Result`:
- Option A: Add an optional `oban_job_id` column to the existing table and make `workflow_id`/`step_id` nullable. Avoids a second table.
- Option B: Keep `Workflow.Result` as-is for workflows; add a new `oban_powertools_job_records` table for non-workflow jobs.

**Recommendation: Option B.** The existing `Workflow.Result` has a `unique_constraint([:step_id, :attempt])` that is semantically correct for workflow retry cycles. Jobs have their own attempt counter tracked by Oban. Merging the schemas creates a nullable-FK ambiguity with no benefit. Two focused tables with a shared private formatter function is cleaner. The v1.8 Batch milestone will also need output recording — it can use the same `oban_powertools_job_records` table.

**Call site:** Workers call a `record_output/2` function (or via a return convention like `{:ok, result, output_meta}` from `process/1`) to persist a `JobRecord`. The `perform/1` wrapper handles persisting the result after `process/1` returns.

**Return value convention:** Add `{:ok, result, record_opts}` as a new valid return from `process/1`. `record_opts` is a keyword list accepted by `record_output`. This is additive and does not break existing workers — existing `:ok` and `{:ok, value}` returns continue to work unchanged.

### 4. redact: At-Rest (Drop at Persist)

**Existing seam:** `Workflow.Result` has a `redacted` boolean column. The existing `DisplayPolicy.workflow_result/2` function already checks this flag to hide payload in the UI. The `Redactor` behaviour mentioned in the milestone context refers to the `DisplayPolicy` + `render_job_field` pattern; there is no standalone `Redactor` module in the codebase as of v1.6.

**Integration point for redact: at worker declaration:** `use ObanPowertools.Worker, redact: [:card_number, :ssn]` declares fields whose values should be dropped from the persisted `JobRecord.payload`. The implementation is: at `record_output` time, remove (not encrypt) the declared keys from the payload map before writing to `oban_powertools_job_records`. The `redacted` boolean is set to `true` if any field was stripped.

**Integration point for display:** The `DisplayPolicy.render_job_field/3` function gains a new kind: `:job_recorded`. The existing `:job_args` and `:job_meta` kinds already exist in the display policy contract (confirmed in `HexConsumerWeb.ObanPowertoolsDisplayPolicy`). Adding `:job_recorded` is additive.

**What does NOT change:** The `idempotency_fingerprint` is computed over raw args before any redaction. The redaction happens only at output-record persist time, not at enqueue time. This preserves the existing idempotency semantics.

## New Components

### ObanPowertools.Worker.Hooks (internal module)

A private dispatch module that `ObanPowertools.Worker` uses to call declared hooks. Not a public behaviour — hooks are optional callbacks on the worker module itself (`on_start/1`, `on_success/2`, `on_failure/2`, `on_discard/2`). Workers opt in by implementing the callbacks; `defoverridable` provides no-op defaults. The dispatch module wraps each call in a `try/rescue/catch` with a `Logger.warning/2` on crash. No registry, no GenServer.

### ObanPowertools.JobRecord (Ecto schema)

New schema for `oban_powertools_job_records`:

```
id (uuid, pk)
oban_job_id (integer, not null, index — references oban_jobs.id but no FK to avoid coupling)
worker (string, not null, index)
attempt (integer, not null, default 1)
status (string, not null, default "ok")
payload (map, not null, default {})
payload_bytes (integer, not null, default 0)
retention (string, not null, default "standard")
redacted (boolean, not null, default false)
summary (string, nullable)
recorded_at (utc_datetime_usec, not null)
expires_at (utc_datetime_usec, nullable)
inserted_at (utc_datetime_usec, autogenerate)
```

Index: `(oban_job_id, attempt)` unique. Index on `worker`. Index on `status`. Index on `expires_at`.

No FK to `oban_jobs` in the migration — Oban prunes jobs from its own table; a hard FK would block pruning. The `oban_job_id` is a soft reference only.

### Migration: oban_powertools_job_records

New migration added to the Igniter installer and the example host. Follows existing migration naming conventions (prefix `20260522...` series).

## Modified Components

### ObanPowertools.Worker (extended)

Changes:
1. Strip `:deadline`, `:timeout`, `:redact`, `:hooks` (or individual `on_start`/etc.) from `oban_opts` before passing to `use Oban.Worker`.
2. Compile-time: if `timeout: N` declared, generate `def timeout(_job), do: N` override.
3. Compile-time: if `deadline: N` declared, store `@powertools_deadline_ms N` and inject a deadline check at the top of `perform/1`.
4. Compile-time: if `redact: fields` declared, store `@powertools_redact_fields fields`.
5. In `perform/1` wrapper: call `on_start/1` before `process/1`, then call appropriate hook after `process/1` based on return value.
6. In `perform/1` wrapper: if the worker module has `record_output?: true` or the `process/1` return is `{:ok, result, record_opts}`, persist a `JobRecord` using `ObanPowertools.JobRecord.record/3`.

New declared callback specs in the generated module:
```elixir
@callback on_start(Oban.Job.t()) :: :ok | term()  # optional
@callback on_success(Oban.Job.t(), term()) :: :ok | term()  # optional
@callback on_failure(Oban.Job.t(), term()) :: :ok | term()  # optional
@callback on_discard(Oban.Job.t(), term()) :: :ok | term()  # optional
```

All are `defoverridable` with no-op defaults so existing workers do not break.

### ObanPowertools.DisplayPolicy (extended)

Add `:job_recorded` as a recognised kind in `render_job_field/3`. Document it in the display policy contract. Update the example host `ObanPowertoolsDisplayPolicy.display/3` to handle the new kind.

### ObanPowertools.RuntimeConfig (no change required)

No new host config keys needed for v1.7. Worker hooks are declared on the worker module. `redact:` is declared on the worker module. No global configuration is needed.

### ObanPowertools.Telemetry (additive, frozen contract extension)

Add a `worker` family to the frozen `@contract` for lifecycle hook events. Low-cardinality metadata keys: `[:event, :worker, :outcome]` where `event` is `on_start | on_success | on_failure | on_discard` and `outcome` is `ok | crash_caught`. This preserves the low-cardinality invariant.

New `execute_worker_event/3` function matching the existing pattern. Telemetry emission happens inside the crash-catching wrapper so hook crashes still emit telemetry.

### mix oban_powertools.install (extended)

Add new migration file generation for `oban_powertools_job_records`. Igniter adds it to the migration list in the correct order.

## Data Flow Changes

### Lifecycle Hook Invocation Flow

```
Oban.Queue.Executor.perform/1
  → worker.perform(%Oban.Job{})        # generated by ObanPowertools.Worker
      → maybe_check_deadline(job)       # returns {:discard, :deadline_exceeded} if past
      → call_hook(:on_start, job)       # crash-caught, result discarded
      → process(typed_job)              # worker's business logic
      ↓ return value from process/1
      → dispatch_hooks(job, result)     # on_success | on_failure | on_discard, crash-caught
      → maybe_record_output(job, result) # if record_opts present or auto-record enabled
      → return result to Oban executor
```

The hook dispatch logic:
- `{:ok, _} | :ok` → `on_success/2`
- `{:error, _}` → `on_failure/2` (job will retry if attempts remain)
- `{:discard, _} | :discard` → `on_discard/2` (job is discarded permanently)
- `{:cancel, _}` → treated as discard for hook purposes (fire `on_discard/2`)
- Exception/crash → `on_failure/2` (Oban catches the crash above; hooks fire before re-raise)

### Output Recording Pipeline

```
process/1 returns {:ok, result, [payload: %{...}, summary: "...", redact: true]}
  → perform/1 intercepts the three-tuple
  → strips redact: fields from payload if @powertools_redact_fields declared
  → calls ObanPowertools.JobRecord.record(repo, job, attrs)
      → inserts into oban_powertools_job_records
  → returns {:ok, result} to Oban executor (strips the record_opts from return)
```

The `record/3` function is fault-tolerant: if the DB write fails, a `Logger.warning/2` is emitted but the job outcome is not affected. Output recording failure never causes a job retry.

Redaction at persist:
```
attrs.payload
  |> Map.drop(@powertools_redact_fields)   # drop by key name
  → stored as payload with redacted: true
```

The raw payload never touches the database. The drop happens in the `perform/1` wrapper before `record/3` is called.

### Workflow.Result Impact

No change to `Workflow.Result`. The `Runtime.complete_step/3` call path continues to write to `oban_powertools_workflow_results`. The generalised output recording is a separate table and call path. There is no migration of existing workflow result rows.

The `Workflow.Step` still has `has_many(:results, Workflow.Result)`. The new `JobRecord` is not associated from `Workflow.Step` — workflow steps are the unit of record for the workflow layer.

### DisplayPolicy: job_recorded Kind

```
JobsLive (job detail view)
  → loads JobRecord for the job (if present)
  → DisplayPolicy.render_job_field(:job_recorded, record_input, %{job: job})
      → host implements display(:job_recorded, input, context) or falls back to default
  → renders summary / payload in job detail UI
```

`record_input` is a map with the same keys as `Workflow.Result.display_input/1` for consistency. This allows host display policies to use the same pattern for both workflow results and job records.

## Suggested Build Order

### Phase 1: Worker Hooks (on_start / on_success / on_failure / on_discard)

**Rationale:** Self-contained change entirely within `ObanPowertools.Worker`. No schema migration, no new table, no display changes. Establishes the lifecycle wrapper that all subsequent phases depend on. The `perform/1` wrapper refactoring done here is the foundation for deadline checking (Phase 2) and output recording (Phase 3).

**Deliverables:**
- Crash-caught hook dispatch in `perform/1`
- `on_start/1`, `on_success/2`, `on_failure/2`, `on_discard/2` optional callbacks with no-op defaults
- Telemetry emission for hook invocations (additive `worker` family)
- Tests: hook called on each outcome, crash in hook does not change job result, telemetry emitted

### Phase 2: deadline: / timeout: Pass-through

**Rationale:** Compile-time only changes to the worker macro (deadline check in `perform/1`, `timeout/1` generation). Depends on the `perform/1` wrapper refactoring from Phase 1 (deadline check slot is defined there). No schema changes.

**Deliverables:**
- `timeout: N` compiles to `def timeout(_job), do: N` override
- `deadline: N` compiles to `@powertools_deadline_ms N` + top-of-`perform/1` guard
- Strip `:deadline` and `:timeout` from oban_opts before passing to `use Oban.Worker`
- Tests: timeout respected, deadline exceeded returns `{:discard, :deadline_exceeded}`

### Phase 3: Output Recording (JobRecord + record_opts return convention)

**Rationale:** Needs the Phase 1 `perform/1` wrapper to intercept the three-tuple return. New schema table, new migration, Igniter installer update. No dependency on Phase 2.

**Deliverables:**
- `oban_powertools_job_records` migration
- `ObanPowertools.JobRecord` Ecto schema
- `{:ok, result, record_opts}` return convention from `process/1`
- `JobRecord.record/3` fault-tolerant persist
- Tests: record written on `{:ok, result, record_opts}`, failure to write does not affect job outcome

### Phase 4: redact: At-Rest

**Rationale:** Depends on Phase 3 (output recording pipeline). `redact:` fields are stripped from `record_opts.payload` at persist time. The `@powertools_redact_fields` module attribute is compiled in Phase 4 as an extension to Phase 3's `perform/1` interceptor.

**Deliverables:**
- `redact: [:field1, :field2]` compile-time option, strips `:redact` from oban_opts
- Payload field dropping before `JobRecord.record/3`
- `redacted: true` flag written to `JobRecord`
- `:job_recorded` display kind in `DisplayPolicy`, example host updated
- Tests: declared fields absent from persisted payload, `redacted: true` set, non-declared fields preserved

### Cross-Cutting: Migration Installer

Add `oban_powertools_job_records` migration to Igniter installer and `examples/phoenix_host`. This is logically part of Phase 3 but can be parallelised with Phase 4 since the schema is stable after Phase 3.

### Phase Ordering Rationale

```
Phase 1: Worker Hooks
  → establish perform/1 wrapper (required by all phases)
  → no schema dep

Phase 2: deadline: / timeout:
  → compile-time only
  → depends on Phase 1 wrapper shape
  → no schema dep

Phase 3: Output Recording
  → schema dep (new table)
  → depends on Phase 1 wrapper (intercept return)
  → independent of Phase 2

Phase 4: redact: at-rest
  → depends on Phase 3 (record_output pipeline)
  → schema already in place from Phase 3
```

Phases 2 and 3 can be built concurrently if needed (they touch different parts of the macro). Phase 4 is strictly after Phase 3.

## Key Constraints Carried Forward

- **Telemetry contract remains frozen:** New `worker` family is additive. Do not remove or rename existing family keys.
- **perform/1 return convention is additive:** `{:ok, result, record_opts}` is new; `:ok`, `{:ok, value}`, `{:error, _}`, `{:cancel, _}`, `{:snooze, _}`, `:discard`, `{:discard, _}` continue unchanged.
- **No FK to oban_jobs in job_records migration:** Oban prunes its own table; hard FK blocks pruning.
- **Hook crashes are never fatal:** `rescue` + `catch` + `Logger.warning/2` is the contract.
- **Idempotency fingerprint is unchanged:** Fingerprint is computed over raw args before any redaction, at enqueue time. No v1.7 feature touches `generate_fingerprint/2`.
- **DisplayPolicy host contract is additive:** Hosts that do not implement `display(:job_recorded, ...)` get a safe default. The existing policy pattern falls through to `nil` → default rendering.
- **Batches milestone (v1.8) reuses these seams:** Worker hooks and JobRecord are the shared infra that v1.8 batch callbacks and batch output recording will build on. Do not gate v1.8 features into the v1.7 schema.
