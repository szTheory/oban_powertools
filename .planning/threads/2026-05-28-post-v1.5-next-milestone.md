# Post-v1.5 Adopter Readiness & Next-Milestone Ordering

Date: 2026-05-28
Status: Active — supersedes `2026-05-25-adopter-readiness-and-v1.3-ordering.md`

## Question

After v1.5 (Native Job Surface & Automation API) shipped: how close is Oban Powertools to "done enough"
for its scope, and what is the single highest-leverage next milestone?

## Method

Repo-grounded assessment (3 read-only Explore agents over planning history, `prompts/` vision, and
`lib/` source reality) + 3 deep research agents (Batches, Worker-Lifecycle, Observability/Health).
Conclusions are grounded in actual source/tests/shipped evidence, not milestone names.

## Repo-grounded answer

**Done-%: ~87% (band 80–89, "strong, meaningful wedges remain"), trending toward 90+ for the narrow
scope.** The whole operator control plane is real and coherent: typed workers, idempotency, limiters,
cron, durable workflows, Lifeline repair, forensics/runbooks, native `/ops/jobs` browse/detail/actions
(single + bulk), and the `Operator` Elixir API — one Lifeline pipeline, frozen low-cardinality
telemetry, explicit support-truth. 40 test files, 4 CI lanes, 13 guides.

**The foundational gap is not a feature: the lib is `0.1.0` and unpublished on hex after 5 milestones.**
For a serious Phoenix SaaS adopter, that's the single biggest barrier to real value — bigger than any
missing capability. Decision captured this pass: **hex publication is a near-term goal.**

## Recommended ordering

1. **v1.6 Release & Operability** *(THE pick)* — first public hex release at a deliberate `0.x` (recommend
   `0.5.0`; document a path to `1.0` *after* real adopter feedback) + `mix oban_powertools.doctor`
   (index/invalid-index/uniqueness-timeout/config/migration-drift, read-only `pg_catalog`, exit codes)
   + `mix oban_powertools.limiter.explain`/`.simulate` (CLI over existing `Explain`+`Limits`, ships the
   rate-limit glossary) + Parapet/SLO telemetry guide & opt-in `Telemetry.metrics/0` over the frozen
   contract (NO `oban_met` dep) + getting-started verified from hex. Zero new deps, near-zero runtime
   risk, ships two named footguns from `prompts/oban_powertools_context.md`, unblocks ALL adopter value.
2. **v1.7 Worker Lifecycle & Safety** — hooks (`on_start/success/failure/discard`, **observe-only,
   crash-caught, never change job outcome**), soft `deadline:` (pre-run cancel) + `timeout:` pass-through
   to Oban's own `timeout/1` (no custom BEAM task machinery), output recording by **generalizing
   `Workflow.Result`** into a shared recordings table keyed by `oban_job_id`, and `redact:` at-persist.
   **Defer `encrypt:`.**
3. **v1.8 Batches & Composition** — dedicated `batches`/`batch_jobs` tables (NOT a 10k-node DAG),
   `completed` + `exhausted` callbacks via the **generalized existing callback outbox + dispatcher**,
   opt-in ack mix-in, chains as thin linear-DAG sugar over `Workflow.insert/2`, native Batches page with
   Lifeline-routed bulk-retry of failed members. Defer chunks/nested/growable batches/per-job-progress.
4. **v1.9 Observability / live counts (QRY-06)** — `oban_met` as an *optional* read source gated like
   `oban_web`, never a hard dep, never a metric label.
5. **Native job-surface polish** — QRY-05 (args/meta filter), QRY-07 (Lifeline→job deep-link), QRY-08
   (cross-page select-all), API-03 (`Operator.list/2`). Opportunistic; not its own milestone.

**Ordering rationale (hard dependency):** Worker-Lifecycle **before** Batches — batch callbacks reuse the
hook contract and output recording reuses the same generalized `Workflow.Result` table. Batches-first
forces a later refactor.

## Condensed research conclusions

**Batches:** model a homogeneous batch as dedicated lightweight tables, NOT as the workflow DAG engine
(10k steps + edges + reconcile per ack = wrong at scale). Reuse the workflow engine's *outbox +
dispatcher*, not its step/edge/reconcile machinery. Adopt Oban Pro's honest `completed` (all succeeded,
may never fire) vs `exhausted` (all terminal, always eventually fires) split. **#1 footgun:** hot
counter-row contention + Sidekiq-style negative counts — track per-job terminal state in `batch_jobs`
rows (row-local, no contention); only the ack that flips the last job touches the batch row + inserts a
deduped event. Completion ≠ last job's `perform`. Chains = linear-DAG sugar over the existing engine.

**Worker-Lifecycle:** hooks run inside the executor process but outside any Powertools transaction;
**hook failure NEVER changes the job outcome** (catch+log+telemetry only) — reject Oban Pro
`before_process`/ActiveJob halting semantics. `on_discard` honestly only covers retry-exhaustion (operator
discards already audited via Lifeline). Reuse Oban's `timeout/1` for hard timeouts; soft `deadline:` is a
cheap pre-run check. Output recording generalizes the existing `Workflow.Result` (payload_bytes/summary/
redacted/retention already exist); small byte cap (not Oban Pro's 64MB default). **Defer `encrypt:`** — it
collides with the args-hashing idempotency fingerprint, blinds the v1.5 job filter (encrypted args not
searchable), leaks via meta/errors/stacktraces, and has no proven demand vs the locked "encrypt only when
requested" prior. Ship `redact:` (at-persist field drop via the existing `Redactor` seam) now.

**Observability:** the 4 candidates DON'T form one milestone — they split into Powertools-internal
read-only tooling (doctor, limiter CLI, telemetry guide: low-risk, zero-dep, **core**) vs generic live
counts (oban_met: high-risk dep-creep, **defer**). `doctor` is the highest-leverage zero-dep wedge and
the literal product feature the context doc demands ("ship index health as part of doctor"). Live counts
belong on the QRY-06/UI track with `oban_met` as an *optional* read source, because generic job/queue
counts are exactly what the Oban Web bridge already covers — adopting oban_met into a health milestone
drifts toward "rebuild Oban Web." The native overview's value is the diagnosis-bucket surface, not counts.

## Overbuilding verdict

The maintainer is at **mild risk of overbuilding**. The feature set already exceeds what most solo/indie
Phoenix apps use; the lib is invisible (unpublished). Build the **release**, not another capability;
then let real adoption decide whether Batches/hooks are wanted. After v1.8, expect diminishing returns
for the narrow scope.

**Defer-until-signal (don't build yet):** `encrypt:` (until requested), prioritizer/autoscaler (until
adoption proves demand), nested batches/chunks/growable batches (Sidekiq's worst reliability area),
`oban_met` as a hard dep / native metrics dashboard (= rebuild Oban Web), mobile/companion surfaces,
generic event bus.

## Graduation candidate (cross-phase process rule)

From v1.5: **audit/verify must assert a clean working tree (or per-phase commit existence), not validate
working-tree state.** v1.5 phases 44/45 audited `passed` while uncommitted. This is a cross-phase GSD
process rule worth graduating into the audit/verification contract, not just a one-off `RETROSPECTIVE`
note.

## Explicit non-priorities (carried forward)

- Rebuilding the generic Oban Web job/queue UI — the bounded read-only bridge is the honest answer.
- Non-Postgres coordination layers; presenting workflows as a general orchestration platform.
