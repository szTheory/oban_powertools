# Phase 52.1: Close gap: REL-04 — fix verify-published CI (Igniter committed-modules conflict) - Context

**Gathered:** 2026-05-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the blocker that prevents the `verify-published` CI job from completing end-to-end. The `examples/hex_consumer/` committed scaffold includes `oban_powertools_auth.ex` and `oban_powertools_display_policy.ex` — files that `mix oban_powertools.install` tries to create via `Igniter.Project.Module.create_module`. When they already exist, Igniter emits `:error issues` and exits 0 without generating any migration files. The subsequent `mix ecto.migrate` runs with 0 migrations; `priv/repo/seeds.exs` then fails inserting into tables that were never created — the first-session test never runs.

Fix scope: one blocker fix (Igniter conflict), one housekeeping item (gitignore `hex_consumer_local_proof`), two CI cleanup items (MIX_ENV consistency, sed robustness).

No new library capability. No changes to the hex consumer's first-session test or seeds.

</domain>

<decisions>
## Implementation Decisions

### Igniter conflict fix
- **D-01:** Fix mechanism is a **pre-install `rm -f` step** in the `verify-published` job in `.github/workflows/release.yml`. Immediately before the "Run installer from published tarball" step, add a step that deletes:
  - `examples/hex_consumer/lib/hex_consumer_web/oban_powertools_auth.ex`
  - `examples/hex_consumer/lib/hex_consumer_web/oban_powertools_display_policy.ex`
  This allows Igniter to generate them fresh from the published tarball, which then also generates the 6 required migration files. The committed scaffold is unchanged — local dev continues to work without needing to run install.
- **D-01a:** Also add a comment to `examples/hex_consumer/regenerate.sh` explaining that `oban_powertools_auth.ex` and `oban_powertools_display_policy.ex` are committed as local dev scaffolding but are overwritten by `mix oban_powertools.install` on each fresh install.

### hex_consumer_local_proof cleanup
- **D-02:** Add `examples/hex_consumer_local_proof/` to `.gitignore`. It is a temp local dev artifact (created during Phase 51 iteration) and should not appear in `git status`. Do not commit it; do not delete it — the owner decides when to clean up their local working tree.

### CI info-level fixes (both in scope for this phase)
- **D-03:** Add `env: MIX_ENV: test` to the "Compile hex consumer" step in the `verify-published` job. This eliminates the recompile triggered when subsequent ecto/seed/test steps switch to `MIX_ENV: test`. One-line change; consistent with all other steps.
- **D-04:** Update the `sed` version-pin command in the `verify-published` job to use a version-agnostic pattern that matches on the dep name rather than hardcoding `~> 0.5`. Suggested pattern: `s|{:oban_powertools, "~> [0-9][0-9.]*"}|{:oban_powertools, "== ${VERSION}"}|` — survives future version bumps to `~> 0.6` or `~> 1.0` without silently no-oping.

### Claude's Discretion
- Exact step name and placement for the pre-install rm step (before vs. after checkout, after cache; "Delete committed installer-generated files" is a reasonable step name).
- Whether to use `rm -f` or `rm` (prefer `rm -f` to avoid non-zero exit if a future scaffold change removes one of the files).
- Minor: can combine D-03 (MIX_ENV) and the rm step if it simplifies the job YAML, but separate named steps are preferable for CI readability.

</decisions>
