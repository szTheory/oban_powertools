## Visuals

### Layout Strategy
Phase 2 stays inside the existing `/ops/jobs` shell and adds four narrow native surfaces:
1. **Overview:** health cards for limiter saturation, blocked jobs, paused cron entries, and missed slots.
2. **Limiters:** a dense index with named limiter resources, scope summary, saturation state, cooldown state, and sampled blocker evidence.
3. **Cron:** a dense index with source badge, overlap policy, catch-up policy, timezone, last run, next run, and pending slot state.
4. **Audit:** a chronological operator trail spanning limiter actions, cron actions, and bridge-originated Oban Web actions.

Blocked-job detail is a side panel or dedicated detail pane, not a separate full job console. It must deep-link to Oban Web for generic job inspection.

### Visual Hierarchy
1. **Current safety state:** blocked/runnable status, active cooldowns, paused entries, and missed-slot warnings sit above all secondary metadata.
2. **Explanation evidence:** live blocker summary, retry timing, and safe next action appear before raw JSON, IDs, or timestamps.
3. **Policy metadata:** overlap policy, catch-up policy, source ownership, partition scope, and algorithm details remain visible but secondary.
4. **History/audit context:** snapshots, prior actions, and historical evidence are tertiary and collapsible.

### Focal Points
- "What is blocked right now?"
- "Why is it blocked?"
- "When will it run again?"
- "What action is safe next?"

### Interaction Contract
- Use an explanation-first detail pattern for blocked jobs:
  live state first, snapshot evidence second, raw structured details last.
- Show explicit **Live Now** and **Snapshot** labels anywhere `explain/1` evidence appears. Never blend them into one panel.
- Render blocker lists in deterministic order with stable code badges, coarse scope label, summary sentence, retry timing, and operator-action availability.
- Limiters table rows must expose one primary entry point: `Inspect Job Blockers`.
- Cron rows must expose at most three visible row actions: `Pause Cron Entry`, `Resume Cron Entry`, and `Run Now`. Overflow anything else behind a secondary menu.
- Every mutating action opens a preview-first confirmation surface that shows impact, actor, reason field, and audit consequence before commit.
- Code-managed cron entries must show a persistent `Code` badge; runtime-managed entries must show a persistent `Runtime` badge.
- Code-managed entries may be paused/resumed and manually triggered, but schedule semantics must render read-only with helper copy explaining that edits belong in code.
- Overlap and catch-up policies render as compact pills in tables and as plain-language sentences in details.
- Saturation and cooldown states must be visible at row level without opening detail.
- Empty, loading, and error states should preserve page structure so operators do not lose context during refresh.

### Data Presentation Rules
- Prefer compact operator language over implementation jargon in visible labels:
  use `Blocked`, `Runnable`, `Cooling Down`, `Missed Latest Slot`, `Queued Follow-Up`.
- Use relative time plus exact timestamp together for next-run and retry-at values.
- Keep high-cardinality values out of badge labels in overview cards; surface them only in detail panels.
- Structured explanation payloads may be shown in a collapsible developer section, but never as the default first view.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Inspect Job Blockers |
| Empty state heading | No smart engine resources yet |
| Empty state body | Sync a code-managed cron entry or add a limiter binding to see live engine state here. |
| Error state | Smart engine state could not be loaded. Refresh the page, then check database connectivity and telemetry logs if the problem persists. |
| Destructive confirmation | Clear Limiter Cooldown: This may release queued work immediately. Enter a reason before continuing. |

Additional labels locked for this phase:
- Live explanation tab: `Live Now`
- Snapshot explanation tab: `Snapshot at Block Start`
- Code-managed source badge: `Code`
- Runtime-managed source badge: `Runtime`
- Overlap default badge: `Queue One`
- Catch-up default badge: `Latest Only`
