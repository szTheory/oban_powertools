# Phase 62 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Phoenix LiveView HEEx with Tailwind utility classes already used by native `/ops/jobs` pages |
| Icon library | none for this phase; existing native pages use text controls, badges, and links |
| Font | Inherit host Phoenix/browser sans-serif stack; do not introduce a bundled font |

Notes:
- Source: existing `JobsLive`, `LifelineLive`, router, and selectors. This is an Elixir/Phoenix LiveView codebase, not React/Next/Vite; shadcn initialization is not applicable.
- Add a native route and selector helper for `/ops/jobs/batches`. Keep the host-owned outer `/ops/jobs` shell and the optional Oban Web bridge boundary intact.
- The UI must feel like an operator console: dense, inspectable, table-first, audit-first, and quiet. Do not use a marketing layout, hero section, decorative imagery, or nested cards.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline badge gaps, compact metadata rows, checkbox-to-label gaps |
| sm | 8px | Button padding axis, table cell micro-gaps, stacked helper text |
| md | 16px | Card padding, filter gaps, modal inner rhythm |
| lg | 24px | Page padding, main panel gaps, table section spacing |
| xl | 32px | Major vertical separation between page header, metrics, and data panels |
| 2xl | 48px | Reserved for rare full-width empty/error states only |
| 3xl | 64px | Not used in this phase |

Exceptions: 40px minimum height for text buttons and inputs; 44px minimum target for checkboxes/selectable rows on touch-width screens; progress bars may use 8px height.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 600 | 1.4 |
| Heading | 16px | 600 | 1.2 |
| Display | 24px | 600 | 1.2 |

Rules:
- Use exactly these four sizes for the Phase 62 UI. Existing `text-2xl`, `text-base`, `text-sm`, and `text-xs` classes map to this contract.
- Use only regular `400` and semibold `600`. Do not add bold, light, condensed, or monospace except inside JSON/error `<pre>` payloads.
- Keep table body text at 14px. Use 12px only for badges, timestamps, compact counts, and secondary diagnostic hints.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Main cards, tables, detail panels, modals |
| Secondary (30%) | `#f8fafc` | Table headers, filter panels, code/payload wells, progress-track backgrounds |
| Accent (10%) | `#4f46e5` | Active tabs, selected rows, primary retry/preview actions, focus rings, selected checkboxes, links |
| Destructive | `#dc2626` | Execute remediation, discard/cancel-style destructive actions, terminal failed callback badges |

Accent reserved for: active status tabs, selected table rows, selected bulk-action banner, primary "Preview Retry" / "Retry Failed Jobs" actions, focus rings, internal links, and progress-fill for healthy in-progress batches.

Semantic colors:
- Warning `#d97706`: retryable, exhausted, callback failed, output unavailable, blocked, drifted preview, read-only notices.
- Success `#059669`: completed batches, delivered callbacks, successful execution flash, healthy callback state.
- Neutral `#475569`: inserting/executing/claimed/scheduled states, secondary buttons, bridge links.

Do not use purple/blue gradients, decorative background colors, or per-row rainbow status palettes. State badges must follow the existing border + pale background + text pattern.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Retry Failed Jobs |
| Empty state heading | No batches match this view |
| Empty state body | No batch rows match the selected status and filters. Try a different status, clear filters, or inspect Jobs for ungrouped work. |
| Error state | Could not load batch data. Refresh the page; if this continues, verify the Powertools batch and callback migrations are installed. |
| Destructive confirmation | Retry Failed Jobs: Review the Lifeline preview, enter a reason, then confirm retry. This writes audit evidence for every attempted job. |

Additional required copy:
- Page title: `Batches`
- Page description: `Inspect batch and chain progress, failed members, blocked states, and Lifeline recovery paths. Native Powertools pages own audited mutations; generic job internals deep-link into the Oban Web bridge.`
- Read-only banner: reuse `LiveAuth.page_read_only_banner(:batches)` once permission naming exists.
- Selected banner: `{N} failed jobs selected`
- Bulk retry modal title: `Retry Failed Jobs`
- Bulk retry modal body: `Lifeline will preview each selected failed job before execution. Jobs that changed state before execution are skipped and reported.`
- Reason label: `Reason (required)`
- Reason placeholder: `e.g., upstream outage resolved, safe to replay failed rows`
- Success flash: `Batch retry complete: {successes} retried, {failures} skipped or failed.`
- Callback retry CTA: `Preview Callback Retry`
- Callback empty copy: `No stuck or dead callbacks are blocking this batch.`
- Blocked explanation heading: `Why this batch is blocked`
- Output-unavailable explanation: `A chain step needs upstream output that is missing, expired, or was not recorded. Review the failed callback and retry only after the upstream output contract is corrected.`

Destructive actions in this phase:
- `Retry Failed Jobs`: gated by Lifeline preview, non-empty reason, preview-token execution, per-job success/failure reporting, and audit evidence.
- `Retry Callback`: gated by Lifeline preview, non-empty reason, preview-token execution, drift handling, and audit evidence.
- No cancel, discard, delete, prune, retention editing, or cross-page select-all action is in scope.

## Phase 62 Layout Contract

### `/ops/jobs/batches` Index

Use a single page wrapper: `space-y-6 p-6`.

Top sequence:
1. Header with `h1` `Batches` and the page description at 14px zinc text.
2. Read-only amber banner when mutations are unavailable.
3. Four metric cards in `grid gap-4 md:grid-cols-2 xl:grid-cols-4`: `Total Batches`, `Needs Attention`, `Executing`, `Completed`.
4. Status tabs in `flex flex-wrap gap-2`: `all`, `inserting`, `executing`, `exhausted`, `insert_failed`, `callback_failed`, `completed`.
5. Filter form in `flex flex-wrap gap-4`: batch name/id, status, chain-only toggle, queue/worker if available from member joins.
6. Bulk selected banner appears only when failed members are selected. Use indigo background, count on the left, `Retry Failed Jobs` on the right.
7. Table panel: `overflow-hidden rounded-lg border bg-white`.

Table columns:
- Select checkbox, visible only for rows or expanded failed-member entries that are retry-eligible.
- Batch: name or shortened UUID, with `Chain` badge when chain metadata is present.
- Status: badge plus blocked-state hint when applicable.
- Progress: numeric `success_count + discard_count / total_count`, 8px progress bar, and percentage.
- Failed: failed/discarded count and retryable member count.
- Callbacks: pending/failed/claimed/delivered summary with attention badge for stuck/dead callbacks.
- Updated: relative or formatted timestamp using existing `timestamp_copy` style.
- Actions: `Open Batch`, `Retry Failed Jobs` when eligible, `Preview Callback Retry` when a failed callback blocks progress.

Row expansion/detail:
- Clicking a batch row opens a detail page or inline detail panel; choose one implementation pattern, but URL-addressable detail is preferred as `/ops/jobs/batches/:id`.
- Expanded content must show failed members, callback rows, chain step metadata, and blocked explanation without relying on hidden hover-only details.

### Batch Detail

Use `space-y-6 p-6`.

Required sections:
- Header: `Batch {name-or-short-id}`, status badge, `Back to Batches`, primary eligible action.
- Identity card: batch id, name, chain id if any, inserted count, chunk count, total count, created/updated/completed timestamps.
- Progress card: counts, progress bar, status-specific explanation.
- Failed member inspection: table of failed/discarded jobs with job id, worker, queue, state, attempt, last error summary, and `Open Generic Job Inspection in Oban Web bridge`.
- Callback outbox: table of callbacks scoped to the batch with event, status, attempts, available/claimed/lease/delivered timestamps, last error, and retry action where eligible.
- Chain context: show current/last known step, step index/count, upstream job id, output dependency status, and explicit output-unavailable copy when relevant.
- Manual intervention history: reuse existing Lifeline/Audit presentation pattern when audit events are available.

### Modal / Preview Pattern

Reuse the existing Jobs/Lifeline preview contract:
- Fixed backdrop `fixed inset-0 bg-zinc-900/50 backdrop-blur-sm z-50`.
- Modal panel `bg-white rounded-lg shadow-xl p-6 w-full max-w-md`; use `max-w-2xl` only when listing multiple affected records.
- Preview content must show before/after state, affected records, skipped records, audit consequence, preview status, and preview token.
- Execute button stays disabled until reason is non-empty and preview state is executable.
- Drift, consumed preview, expired preview, unauthorized, and reason-too-short errors render in red/warning copy inside the modal or detail panel.

## Interaction Contract

Required interactions:
- Status tabs update the URL query and reset pagination/selection.
- Filters update by LiveView change events and preserve shareable query params.
- Selection is page-local only. Do not add cross-page select-all in this phase.
- Bulk retry selects only retry-eligible failed jobs within the visible batch/member context.
- `Retry Failed Jobs` calls Lifeline preview/execute per selected job and reports successes/failures. It must not bypass Lifeline because BUI-03 explicitly requires Lifeline-routed recovery.
- `Preview Callback Retry` must surface callback event, dedupe key, attempts, last error, and target batch/chain context before execution.
- Callback recovery must make stuck/dead callback states visible before offering execution. Do not hide failed callback rows behind a generic blocked badge.
- Links to generic job details use the existing Oban Web bridge path and copy. Native Powertools pages own audited mutations.
- Empty, read-only, unauthorized, preview drift, and load error states must be visible and copy-complete.

Blocked-state explanations:
- `insert_failed`: show failed chunk, inserted count, total count, and stored insert failure reason.
- `callback_failed`: show the failed callback event, attempts, last error, and whether retry is available.
- `output_unavailable` / `output_expired`: show upstream job id, step name/index, and that the upstream output contract must be corrected before retry.
- `executing but not complete`: show remaining count and whether callbacks are pending, claimed, or unavailable.
- `exhausted`: show discarded count and failed member inspection with retry eligibility.

Accessibility:
- All buttons must be real `<button>` elements with `type="button"` except form submit buttons.
- Checkboxes must have accessible labels or table-header context.
- Selected rows must not rely on color alone; preserve checkbox state and status text.
- Disabled buttons must include adjacent helper copy explaining why.
- Keyboard users must be able to tab through filters, table actions, modal reason input, cancel, and execute controls.
