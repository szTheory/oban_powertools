# Milestone v1.2: Workflow Semantics & Recovery

**Status:** ✅ SHIPPED
**Phases:** 16-23
**Total Plans:** 22

## Overview

Oban Powertools v1.2 hardens workflows from durable DAG primitives into a supportable orchestration substrate. The milestone freezes lifecycle vocabulary, durable cause semantics, DB-first mutation rules, callback and recovery behavior, await/signal/expiry contracts, workflow-local diagnosis, and the proof posture needed to explain in-flight state under real host-app conditions.

## Phases

### Phase 16: Semantics Contract, Cause Vocabulary & Compatibility Baseline

**Goal**: Freeze one explicit workflow and step lifecycle contract before broader runtime changes land.
**Depends on**: v1.1 foundation
**Plans**: 0 plans

**Planned focus:**

- Persist workflow semantics versioning for new rows and define a compatibility path for pre-v1.2 in-flight workflows.
- Introduce durable workflow and step terminal-cause vocabulary.
- Define the legal transition matrix used by runtime, commands, UI, and docs.

### Phase 17: DB-First Transition Engine & Command Pipeline

**Goal**: Ensure all runtime and operator workflow mutations re-enter one legal DB-first path.
**Depends on**: Phase 16
**Plans**: 0 plans

**Planned focus:**

- Centralize cancel, expire, recover, complete, dependency unblock, and failure reconciliation logic.
- Treat PubSub as a hint only; Postgres remains the correctness source of truth.
- Reject unsupported transition attempts with durable operator-visible reasons.

### Phase 18: Durable Callback Outbox & Recovery Attempts

**Goal**: Make workflow callbacks and recovery attempts survive crashes and retries.
**Depends on**: Phase 17
**Plans**: 0 plans

**Planned focus:**

- Add a post-commit callback outbox for terminal and recovery events.
- Persist recovery attempt evidence that preserves already-successful side effects.
- Revalidate semantics and dependency posture before recovery-driven mutation.

### Phase 19: Await Registration, Signal Facts & Expiry Authority

**Goal**: Persist wait/signal truth before layering higher-level operator behavior.
**Depends on**: Phase 18
**Plans**: 0 plans

**Planned focus:**

- Store await registrations with signal identity, correlation, dedupe, and deadline.
- Store incoming signals as durable facts and reconcile them idempotently across arrival timing.
- Establish one authoritative expiry reconciliation path.

### Phase 20: Cancellation, Late Completion & Expiry Semantics

**Goal**: Make cancel, completion, expiry, and late-arrival races explainable.
**Depends on**: Phase 19
**Plans**: 0 plans

**Planned focus:**

- Separate cancel request evidence from final workflow outcome.
- Preserve late completion and late signal evidence instead of hiding it.
- Define precedence rules across cancel, completion, expiry, dependency failure, and late signal arrival.

### Phase 21: Workflow Diagnosis Projection & Native Workflow Surface

**Goal**: Make workflow state operable without database spelunking.
**Depends on**: Phase 20
**Plans**: 0 plans

**Planned focus:**

- Project workflow diagnosis classes from durable workflow/step/wait/signal/recovery evidence.
- Render cause, evidence, and allowed next action in the native workflow screen.
- Render unsupported or unknown states explicitly.

### Phase 22: Lifeline Integration & Bounded Recovery Actions

**Goal**: Unify diagnosis vocabulary and bounded workflow actions across incident and workflow surfaces.
**Depends on**: Phase 21
**Plans**: 0 plans

**Planned focus:**

- Reuse one diagnosis vocabulary between Lifeline and workflow UI.
- Keep workflow recovery/cancel actions bounded, audited, and routed through the workflow command pipeline.
- Keep optional `oban_web` bridge work out of workflow semantics ownership.

### Phase 23: Verification, Upgrade Proof, Telemetry & Support-Truth Closure

**Goal**: Close the milestone with proof and docs that match the actual semantics.
**Depends on**: Phase 22
**Plans**: 3 plans

**Completed focus:**

- Add fixtures for duplicate, late, dropped, pre-await, post-expiry, and race-path events.
- Add archived/in-flight upgrade proof for waiting, retrying, cancelling, and recovering workflows.
- Align telemetry and support-truth docs to low-cardinality markers and proven semantics only.

## Milestone Summary

**Intended outcomes:**

- Workflow lifecycle semantics become explicit, durable, and explainable.
- Callback, recovery, cancel, await/signal, and expiry behavior survive crashes and race conditions without silent meaning drift.
- Native workflow and Lifeline surfaces share one diagnosis vocabulary and one bounded mutation model.
- Public docs and telemetry claim only what the verification stack proves.

**Non-goals:**

- Broad control-plane unification before workflow semantics settle.
- Generic event-bus or webhook platform expansion.
- Full native replacement for commodity Oban Web surfaces.
- Ecosystem-specific automation or provider integrations.
