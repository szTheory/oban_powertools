## Design System

| Property | Value |
|----------|-------|
| Tool | none (source: `30-RESEARCH.md`; Phoenix LiveView surface, shadcn gate not applicable) |
| Preset | not applicable |
| Component library | none (source: codebase inspection of `lib/oban_powertools/web/*.ex`) |
| Icon library | none declared; text labels and badges remain primary (source: codebase inspection) |
| Font | Tailwind default sans stack / system sans (default; no repo font override detected) |

Primary focal point: on every native detail surface, the shared opening-story block is the visual anchor. The diagnosis sentence is the first full-width prose element the eye should land on, with the status badge as the compact scan cue above it and evidence panels visually subordinate beneath it.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Badge text padding, inline icon/text gaps, dense evidence rows |
| sm | 8px | Copy separation inside cards, stacked label/value rhythm |
| md | 16px | Default cell padding, paragraph spacing, action row spacing |
| lg | 24px | Detail-card padding and list-to-detail gutter |
| xl | 32px | Section separation between header, table, and selected-detail pane |
| 2xl | 48px | Major page breaks between overview blocks and deep evidence |
| 3xl | 64px | Reserved for page-level top/bottom breathing room only |

Exceptions: none. Keep action controls at 40px minimum height and preserve the shared opening stack as one uninterrupted block before evidence.

---

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 16px | 600 | 1.3 |
| Display | 24px | 600 | 1.2 |

Use exactly these two weights across the phase: regular `400` and semibold `600`. Status badges stay compact scan elements; diagnosis sentences use Body styling and must appear before any secondary metadata sentence.

---

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` | Primary page background, selected detail surfaces, table bodies |
| Secondary (30%) | `#F8FAFC` | Table headers, preview/evidence panels, secondary cards, read-only framing surfaces |
| Accent (10%) | `#4F46E5` | Selected-row CTA, status-emphasis badges, audit follow-up links, bridge-handoff links |
| Destructive | `#DC2626` | Execution warnings, mutation failures, destructive confirmation emphasis only |

Accent reserved for: review/open diagnosis buttons, selected status badges, `Open in Audit`, and explicit bridge or native venue links. Do not use accent for every interactive control; neutral borders remain the default for read-only navigation and table review buttons.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Review Diagnosis` for read-only surfaces; keep preview-first mutation buttons verb-led and resource-specific, e.g. `Pause Entry`, `Resume Entry`, `Preview Repair` |
| Empty state heading | List-level empty: `Nothing needs review right now` where durable state is healthy; resource-specific empty may stay explicit (`No smart engine resources yet`, `No workflows yet`) if it names the missing source truth |
| Empty state body | Detail-level empty: `Select a row to open diagnosis, next action, venue, and evidence.` List-level empty follow-up: explain what durable signal would make the resource appear or direct the operator back to overview/native source |
| Error state | `Current durable state changed or this action is not available right now. Refresh the detail pane, review the latest diagnosis, and retry only if the next action still applies.` |
| Destructive confirmation | `Pause cron entry`: `This writes an audited pause for the selected entry. Add a reason before confirming.`; `Execute repair`: `This writes a repair execution record for the selected incident. Confirm only after reviewing diagnosis, venue, and audit consequence.` |

Additional copy rules locked for this phase:
- Every selected-resource pane must open in this order: `status badge -> diagnosis sentence -> next action -> venue -> evidence`.
- The first sentence must answer "what is happening and why?" from durable facts. Never start with `Operator Status:`.
- Limiters are diagnosis-only. Replace action-shaped table language with review/open diagnosis language.
- Audit remains read-only and should read as the same control plane story, not as a generic event log.
