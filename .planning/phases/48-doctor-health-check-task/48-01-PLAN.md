<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flags → SQL | `--prefix` (and prefix derived from app env) flows into pg_catalog/information_schema queries as a value crossing into the DB |
| DB → process | Catalog query results are read into the BEAM and rendered downstream (Plan 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-48-01 | Tampering | prefix value in `Checks.*` SQL | mitigate | Bind every prefix/schema value as `$1` and table arrays as `$2` via `repo.query/3`; no `#{}` interpolation of CLI/env input into SQL strings (RESEARCH Pitfall 2) |
| T-48-02 | Information Disclosure | catalog queries reading wrong schema | mitigate | Always filter `n.nspname = $1` / `table_schema = $1`; never list catalog without a namespace filter; Powertools tables pinned to `'public'` |
| T-48-03 | Repudiation/Integrity | cannot-run silently passing a gate | mitigate | `{:error, reason}` from `repo.query/3` becomes a `:error`-severity cannot-run Finding (D-06); never returns `[]` (clean) on query failure |
| T-48-04 | Elevation/Side-effect | accidental Oban start or write path | mitigate | No `@requirements`, no `Oban.start_link`, no INSERT/UPDATE/DELETE; all five checks are SELECT-only over pg_catalog/information_schema/oban_jobs (D-09/D-10) |
</threat_model>
