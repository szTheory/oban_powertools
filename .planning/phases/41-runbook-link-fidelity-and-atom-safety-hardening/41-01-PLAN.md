## Verification gate corrections

Per the orchestrator's resolution of RESEARCH Open Question #4, the WR-01 verification rg-check in CONTEXT.md D-23 is **broken** — it matches 12 lines of legitimate safely-encoded test code today. The researcher's first correction was *also* broken: the pattern `incident_fingerprint=#\{[^}]*\}` scoped to `test/` matches `URI.encode_www_form(...)` interpolations in test assertions (the character class `[^}]*` permits parentheses), so it returns 12 hits today, not 0.

The static rg check is fundamentally wrong-scoped: test code legitimately exercises encoded URL strings in assertions like `"incident_fingerprint=#{URI.encode_www_form(...)}"`. The hazard the gate is meant to catch — raw fingerprint interpolation that breaks routing — only matters in production paths (`lib/`), not in `test/`.

**Corrected (lib-scoped negative-filter) WR-01 gate:**

```bash
rg -n 'incident_fingerprint=#\{' lib/ 2>/dev/null | rg -v 'encode_(www_form|query)' | wc -l
# Must return 0 — matches raw fingerprint interpolation in lib/ that is NOT wrapped in an encoder
```

Empirically verified against HEAD: this returns 0 today (no raw interpolation in lib/ — all lib/ callsites either use `URI.encode_query` directly or build params before encoding). After Phase 41 it must remain 0. The negative-filter pipeline catches any lib/ callsite that interpolates a fingerprint into a URL without wrapping it in `URI.encode_www_form` or `URI.encode_query`. Behavioral coverage in Task 7 (round-trip decode assertions) is the load-bearing proof; the rg check is a hygiene gate against future regressions.

The two earlier candidate patterns (the original D-23 pattern and the test-scoped `incident_fingerprint=#\{[^}]*\}` pattern) are documented here as historical context but are not referenced for execution.
