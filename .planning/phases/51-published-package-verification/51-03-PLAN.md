<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → hex.pm | `verify-published` fetches the published tarball from hex.pm over the network (untrusted external fetch into a CI runner). |
| CI runner → release version output | The `version` flows from the `release-please` job output into a `sed` expression patching `mix.exs`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-05 | Tampering | `:files` whitelist loosened to make the CI job go green | mitigate | Task 2 verify gates that the root `mix.exs` `:files` list is byte-for-byte the verified value; a tarball-content failure is fixed-forward (D-06), never papered over by widening `:files` (D-02, Pitfall 1). |
| T-51-06 | Information Disclosure | secrets leaking into the verify-published job | mitigate | Job declares `permissions: contents: read` and uses NO secrets — it only reads from hex.pm. `HEX_API_KEY` stays scoped to `publish-hex`; it is not referenced in `verify-published`. |
| T-51-07 | Tampering | malicious `version` output injected into the `sed` pin | accept | `version` originates from the trusted `release-please` action (already trusted to cut the GitHub Release + tag); the runner is already privileged to publish. No new trust granted beyond the existing pipeline. |
| T-51-SC | Tampering | hex/mix package resolution from hex.pm in CI | mitigate | The job resolves the exact `== <version>` just published by this same pipeline (D-02b) on a runner gated behind `needs: [publish-hex]` and `release_created == 'true'`; no arbitrary third-party package install is introduced beyond the already-audited consumer deps. |
</threat_model>
