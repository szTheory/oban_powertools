## Visuals

### Layout Strategy
Phase 4 stays inside the existing `/ops/jobs` shell and adds one native Lifeline surface with four tightly scoped operator zones:
1. **Overview strip:** executor health, orphan candidates, pending repair previews, and pruner/archive freshness.
2. **Incident index:** a dense table of heartbeat and orphan events sorted by severity, then newest detection time.
3. **Repair detail:** a master-detail pane showing incident evidence, dry-run output, required reason capture, and execution readiness.
4. **Archive and audit evidence:** read-only panels for immutable intervention history and archive-before-delete visibility.

Do not build a generic jobs console here. Job internals still deep-link to Oban Web. Phase 4 owns Powertools-specific lifeline, repair, and archival state only.

### Visual Hierarchy
1. **Current operational risk:** stale heartbeat age, orphan impact count, and whether human action is required must appear before all secondary metadata.
2. **Dry-run evidence:** the proposed state changes and affected scope must appear before the execute control.
3. **Audit certainty:** actor, reason, preview timestamp, and immutable event id must remain visible anywhere a manual intervention is shown.
4. **Retention posture:** archive and prune freshness is important but always secondary to active incidents and repair decisions.

### Focal Points
- "Is this executor actually dead or just long-running?"
- "What records will this repair change?"
- "Has the situation drifted since I generated the preview?"
- "Was this intervention archived and audited correctly?"

### Interaction Contract
- Default the landing page to active incidents, not archive history.
- Detect and label incidents using heartbeat evidence, never job runtime alone. The UI must always show `Last heartbeat` and `Detection basis` together before any orphan label.
- Use a master-detail pattern on desktop: incident table left, selected incident and repair detail right. On smaller screens, collapse to a stacked flow with the selected incident pinned above the preview sections.
- Every incident row exposes exactly one primary action: `Preview Repair Plan`. Secondary actions may include `View Audit Trail` and deep links into Oban Web, but not direct execution from the table.
- The dry-run surface is mandatory and split into four ordered sections:
  1. `Detection Summary`
  2. `Proposed State Changes`
  3. `Affected Records`
  4. `Audit Record to be Written`
- Show proposed state changes as operator-readable before/after rows first. Raw SQL, raw `Ecto.Multi` steps, or structured payload dumps belong in a collapsed developer section only.
- Any icon-only control must expose an accessible name and a visible text affordance on focus/hover, such as tooltip copy matching the action label for disclosures, expanders, and preview-step toggles.
- Require a written reason before the final execute action becomes enabled.
- The final execute action must remain disabled whenever the incident snapshot has drifted since preview generation. Render this as `Preview Drifted` and require the operator to rerun preview.
- Repair previews are scoped and explicit. Never show a vague `repair all` action. The incident detail must always state the exact workflow, job, executor, or archive batch under review.
- Immutable audit history renders inline for the selected incident and in the shared audit page. Operators should not leave the repair flow just to confirm whether a prior action ran.
- Archive/pruner visibility is read-only in the main incident view. Any prune/archive mutation follows the same preview -> reason -> execute posture as repair actions.
- Empty, loading, and error states must preserve the shell structure and section headings so operators do not lose incident context during refreshes.

### Data Presentation Rules
- Use compact operator language in visible labels: `Healthy`, `Heartbeat Late`, `Executor Missing`, `Orphan Candidate`, `Preview Ready`, `Preview Drifted`, `Archived`, `Pruned`.
- Show relative time plus exact UTC timestamp together for last heartbeat, preview generated at, audit written at, archive completed at, and next prune run.
- Keep overview cards low-cardinality: counts and coarse state only. High-cardinality evidence such as executor ids, workflow ids, node names, and job ids belongs in table rows and detail panels.
- Show affected-scope counts before listing raw record ids, for example `3 workflows`, `12 jobs`, `1 executor`.
- Status color should follow meaning, not novelty: destructive red only for irreversible execution or prune confirmation, amber for warning states, indigo for active selection and preview readiness.
- Audit rows must display actor, action, resource, reason, and immutable event timestamp in the default row view. Expanded metadata is secondary.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Preview Repair Plan |
| Empty state heading | No lifeline incidents need review |
| Empty state body | All tracked executors are heartbeating and no orphan candidates are waiting for repair. Review archive activity below if you need retention evidence. |
| Error state | Lifeline state could not be loaded. Refresh the page, then check heartbeat writes, database connectivity, and audit storage if the problem persists. |
| Destructive confirmation | Execute Repair Plan: This will change job or workflow state immediately and write an immutable audit event. Enter a reason before continuing. |

Additional labels locked for this phase:
- Incident section heading: `Needs Review`
- Healthy section heading: `Healthy Executors`
- Preview drift state: `Preview Drifted`
- Preview success state: `Preview Ready`
- Audit preview section: `Audit Record to be Written`
- Archive visibility section: `Archive Activity`
- Pruner freshness label: `Last Archive + Prune Run`
- Oban Web deep link: `Open Generic Job Inspection in Oban Web`

Destructive actions for this phase:
- `Execute Repair Plan` requires preview completion, written reason, and a non-drifted incident snapshot.
- `Run Archive + Prune Now` requires preview completion, written reason, and explicit affected-count visibility before confirmation.
