## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline badge gaps, severity dot spacing, compact metadata separators |
| sm | 8px | Tight copy stacks inside evidence rows and labels |
| md | 16px | Default card padding, body copy spacing, CTA grouping |
| lg | 24px | Section padding, intro-to-grid separation, card internal section breaks |
| xl | 32px | Gap between major overview regions and desktop column separation |
| 2xl | 48px | Separation between the primary triage zone and secondary continuity/history zone |
| 3xl | 64px | Full-page vertical rhythm when the layout stacks on narrower screens |

Exceptions: none. Reuse the repo's existing `p-6`, `gap-6`, `p-4`, and `space-y-6/8` rhythm.

Interaction density rules:
- Primary overview cards use `p-4` with at least `space-y-3` internally.
- Exemplar rows use `py-3` minimum so the evidence remains scannable without becoming table-heavy.
- CTA rows inside cards keep a minimum 36px vertical hit area.
- Empty and degraded states keep `p-6` so the page shape remains stable during refresh or authorization-driven reduction.

|------|-------|-------|

| Secondary (30%) | `#f8fafc` | Bucket group background, evidence wells, secondary continuity sections, table/list headers where needed |

Accent reserved for: the single highest-priority bucket CTA, the currently selected or emphasized diagnosis bucket, the active destination handoff affordance, and keyboard-visible focus treatment. Do not spend accent on every bucket, every badge, or every plain text link.

Semantic support colors:
- Amber (`#fef3c7` / `#92400e`) for caution, read-only, blocked, or needs-review evidence framing.
- Emerald (`#ecfdf5` / `#047857`) only for resolved continuity evidence and explicitly healthy/runnable reassurance.
- Zinc/slate grayscale for supporting metadata, raw-state labels, and non-primary navigation copy.

## Visuals

### Layout Strategy

Phase 28 replaces the count-heavy overview with a diagnosis-first triage surface:
1. **Orientation header:** one page intro that explains the native-versus-bridge posture in one sentence.
2. **Primary triage grid:** shared operator-status buckets for `Needs Review`, `Blocked`, `Waiting`, `Runnable`, and `Bridge-only Follow-up`.
3. **Secondary continuity block:** `Resolved Recently` appears after active triage, visually quieter and never equal in urgency.
4. **Destination continuity:** each bucket contains bounded exemplar evidence plus one venue-aware next step, rather than dumping the operator into an unscoped list.

Desktop layout rules:
- Use a two-stage composition instead of a single metric grid.
- The first row should privilege the most urgent buckets (`Needs Review`, `Blocked`) before quieter states.
- Cards can span uneven widths if needed to keep diagnosis copy readable; avoid a perfectly uniform KPI wall.

Mobile layout rules:
- Stack cards in priority order.
- Keep diagnosis and venue copy above exemplar evidence so the operator understands the bucket before scrolling rows.
- Preserve CTA visibility without forcing a horizontal scroll region.

### Visual Hierarchy

1. **What needs attention now:** status bucket label plus diagnosis sentence.
2. **Why it matters:** one to three bounded exemplar items with durable facts.
3. **Where to go next:** explicit venue-aware CTA or destination hint.
4. **How trustworthy this is:** ownership badge, inspection-only/native wording, and continuity evidence.
5. **Raw counts:** present, but subordinate to diagnosis and next step.

### Focal Points

- The primary focal point is the `Needs Review` bucket and its next action.
- The secondary focal point is the highest-signal blocked/waiting bucket that explains system friction without implying mutation is available on this page.
- The resolved-history region is intentionally tertiary.

### Interaction Contract

- Overview cards are diagnosis cards, not dashboard vanity cards.
- Every card must answer `what`, `why`, and `where next` before showing more abstract totals.
- Exemplar evidence is representative and bounded; never present it as a complete list.
- Bridge-only destinations must be labeled before click with `Oban Web bridge` and `Inspection only` or equivalent wording.
- Native handoffs must feel like continuation, not restart: the destination loads with the relevant resource already selected when the destination surface supports URL-backed selection.
- Refresh/remount-safe state is URL-owned only for durable context such as selected resource, selected step, active tab, or view.
- The overview itself remains read-only and non-mutating.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Review Needs Review |
| Empty state heading | No operator follow-up needed |
| Empty state body | Powertools does not currently see a native review item, blocked follow-up, or bridge-only inspection path that needs attention. Reopen this page after the next refresh or inspect a specific surface if you are chasing a known issue. |
| Error state | Overview diagnosis could not be assembled from durable state. Refresh the page, then inspect the destination surface directly if the problem persists so you can verify whether the missing context is native or bridge-only. |
| Destructive confirmation | none in Phase 28; the overview remains guidance-only and routes execution to destination surfaces |

Additional locked labels:
- Primary bucket labels: `Needs Review`, `Blocked`, `Waiting`, `Runnable`, `Resolved Recently`, `Bridge-only Follow-up`
- Native ownership label: `Powertools-native`
- Bridge ownership label: `Oban Web bridge`
- Bridge posture label: `Inspection only`
- Native posture label: `Audited action`

Copy rules:
- Diagnosis text must be a factual sentence, not a marketing summary.
- `Next step` copy must name the destination surface, not just say `Inspect`.
- Bridge-only guidance must explain the venue shift before navigation, not after.
- Resolved continuity copy must state an honest time window or source, such as archived repairs or recently closed review items.
- Do not use generic CTA labels such as `Open`, `View`, `Go`, or `Learn More` without a noun.

## Surface Contract

Overview bucket requirements:
- Each active bucket card must include:
  - shared status label
  - subordinate count
  - one diagnosis sentence
  - one to three exemplar evidence rows
  - one venue-aware next-step CTA or destination hint
  - ownership/posture cue when the destination is bridge-only
- Severity may influence exemplar ordering, but the visible bucket title remains the shared operator status.

Exemplar row requirements:
- Each row should show the durable resource or diagnosis subject first.
- Each row should include one short supporting fact such as refusal cause, wait reason, paused state, or repair/archive signal.
- Each row may include a compact destination cue when different exemplars in the same bucket route differently.
- Rows are never editable inline and never expand into a full table on the overview.

Bucket-specific rules:
- `Needs Review` should route primarily to Lifeline or another native diagnosis surface with the relevant item selected.
- `Blocked` and `Waiting` should prefer native Powertools destinations when the diagnosis is Powertools-owned.
- `Runnable` should reassure rather than compete for top attention; if shown, it should still explain what is healthy now.
- `Bridge-only Follow-up` must never borrow native mutation styling.
- `Resolved Recently` belongs below the active triage group, with quieter color treatment and continuity-oriented wording.

Handoff rules:
- Native destinations should use stable URL params for selected row, selected detail, view, or step when those surfaces already support it or will support it in this phase.
- Bridge handoffs should land on an appropriate filtered or scoped bridge destination, not a fake native deep-link contract.
- If a destination cannot preserve exact detail state durably, the copy should promise a scoped list or selected context, not exact replay.

Accessibility and scanability rules:
- Do not rely on color alone for status meaning; labels remain mandatory.
- Keep card headings and CTA labels unique enough for screen reader link lists.
- Ownership badges must remain text-bearing, not icon-only.
