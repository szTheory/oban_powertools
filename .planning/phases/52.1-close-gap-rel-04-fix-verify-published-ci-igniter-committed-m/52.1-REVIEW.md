## Warnings

### WR-01: `OBAN_POWERTOOLS_SKIP_DB_BOOT` has no effect in `verify-published` and misleads maintainers

**File:** `.github/workflows/release.yml:223`
**Issue:** `OBAN_POWERTOOLS_SKIP_DB_BOOT: "1"` is set as a job-level env in the `verify-published` job. This variable is read exclusively by the library's own `test/test_helper.exs` (at the repo root) to skip DB boot during library unit tests. The `verify-published` job runs `mix test` from `working-directory: examples/hex_consumer`, whose `test/test_helper.exs` does not reference this variable at all. The env therefore has zero functional effect in this job. Future maintainers reading the workflow will assume it does something meaningful (e.g., prevents a conflicting DB boot), potentially masking a real misconfiguration or leading them to copy this pattern incorrectly into other jobs.

**Fix:** Remove the `OBAN_POWERTOOLS_SKIP_DB_BOOT` env from the `verify-published` job's `env:` block, or replace it with a comment explaining why it is intentionally absent (if there is a future-proofing reason to keep it). Example removal:

```yaml
  verify-published:
    name: Verify published package (REL-04)
    runs-on: ubuntu-latest
    needs: [release-please, publish-hex]
    if: ${{ needs.release-please.outputs.release_created == 'true' }}
    permissions:
      contents: read
    # Remove: env block with OBAN_POWERTOOLS_SKIP_DB_BOOT — not applicable to hex_consumer tests
    services:
      postgres:
```

## Info

### IN-01: Installer step runs in default (`:dev`) Mix env while compile/test run in `:test`

**File:** `.github/workflows/release.yml:274-276`
**Issue:** The "Run installer from published tarball" step does not set `MIX_ENV`, so it runs in Mix's default `:dev` environment. The immediately following compile step (lines 279-282) explicitly sets `MIX_ENV: test`. This means the installer generates files compiled into the `:dev` build tree (`_build/dev`), and then `mix compile --warnings-as-errors` recompiles them under `:test`. In practice the generated files (`oban_powertools_auth.ex`, `oban_powertools_display_policy.ex`) contain no env-conditional logic, so no breakage occurs today. However, if the installer ever gains env-sensitive generation logic or if the published package adds `:dev`-only deps that the installer references, this silent inconsistency would produce a confusing failure.

**Fix:** Add `MIX_ENV: test` to the installer step for consistency with the rest of the job:

```yaml
      - name: Run installer from published tarball
        env:
          MIX_ENV: test
        run: mix oban_powertools.install
        working-directory: examples/hex_consumer
```

### IN-02: `replace_once` in `regenerate.sh` passes `\n` as literal backslash-n to Ruby's `sub`, producing malformed mix.exs in the regen tree

**File:** `examples/hex_consumer/regenerate.sh:34-37`
**Issue:** The replacement string passed to `replace_once` on line 37 contains `\n` inside a bash double-quoted string. Bash does not expand `\n` in double quotes — Ruby therefore receives the two characters `\` and `n`. Ruby's `String#sub(pattern, replacement_string)` does not treat `\n` as a newline in the replacement string (only back-references `\0`-`\9` and `\&`/`\'`/`` \` `` are special). The resulting `TARGET_DIR/mix.exs` will contain literal `\n` characters instead of newlines, producing syntactically invalid Elixir. The canonical checked-in `examples/hex_consumer/mix.exs` is unaffected (it is maintained by hand), and the script's own TODO instructions require a manual diff before copying, so no CI breakage results. However, anyone running `regenerate.sh` will get a broken intermediate file with no clear error.

**Fix:** Use `$'...'` ANSI-C quoting for the replacement argument so bash expands `\n` to an actual newline before passing it to Ruby:

```bash
replace_once \
  "${TARGET_DIR}/mix.exs" \
  "{:postgrex, \">= 0.0.0\"}," \
  $'{:postgrex, ">= 0.0.0"},\n      {:oban, "~> 2.18"},\n      {:oban_powertools, "~> 0.5"},'
```

Alternatively, use a heredoc or `printf` to construct the replacement string before calling `replace_once`.
