## Summary

The `verify-published` CI job fails because `examples/hex_consumer/` commits two files — `oban_powertools_auth.ex` and `oban_powertools_display_policy.ex` — that `mix oban_powertools.install` tries to create via `Igniter.Project.Module.create_module`. When those files already exist at checkout, Igniter emits `:error issues` and exits 0, silently skipping the 6 required migration files. Without those migrations, `mix ecto.migrate` creates no tables, `priv/repo/seeds.exs` fails inserting into nonexistent tables, and the first-session test never runs.

This phase makes four surgical edits: (1) insert a `rm -f` step before the installer in `release.yml` so Igniter generates files fresh, (2) add a comment to `regenerate.sh` documenting the local-dev vs. CI-install distinction, (3) add `MIX_ENV: test` to the "Compile hex consumer" step to eliminate the recompile triggered when later steps switch environments, and (4) make the `sed` version-pin regex version-agnostic so it survives future minor or major version bumps.

No new libraries. No changes to the committed scaffold files (`oban_powertools_auth.ex`, `oban_powertools_display_policy.ex`). The complete deliverable fits in a single plan.

**Primary recommendation:** Insert the `rm -f` step with exact paths in `release.yml` immediately after the `actions/cache` step and before "Run installer from published tarball". Apply the three remaining edits (D-01a, D-02, D-03, D-04) in the same commit.

The rm step MUST appear after `actions/checkout` (step 1 restores the committed files) and before "Run installer from published tarball" (step 7). The cache step (step 3) is a natural anchor — insert after it.

### Anti-Patterns to Avoid

- **Placing rm before checkout:** Checkout restores the files — rm would be a no-op.
- **Using bare `rm` instead of `rm -f`:** If the scaffold ever removes one of these files from the committed tree, CI would fail with "no such file". `rm -f` exits 0 regardless.
- **Adding `working-directory` to the rm step:** The paths are repo-root-relative; the step runs from the checkout root by default. Do not add working-directory.
- **Hardcoding the version in the sed pattern:** `~> 0.5` silently no-ops when the next minor release changes it to `~> 0.6` or `~> 1.0`. The D-04 regex matches any `~> X.Y[.Z]` pattern.

## Common Pitfalls

### Pitfall 1: Misplacing the rm Step

**What goes wrong:** If the rm step is placed before `actions/checkout`, checkout restores the committed files after the deletion and the rm is a no-op. If placed after "Run installer from published tarball", Igniter has already seen the existing files and emitted errors.

**Why it happens:** Step ordering matters; the committed files are restored by checkout.

**How to avoid:** Anchor the new step immediately before "Run installer from published tarball" (step 7 in the current ordering). The `actions/cache` step (step 3) is a safe "insert after" reference point.

**Warning signs:** CI shows "Igniter: existing modules" or "0 migrations generated" in installer output.

### Pitfall 2: sed Regex No-Op on Future Versions

**What goes wrong:** The existing `~> 0.5` literal in sed silently no-ops when the dep is updated to `~> 0.6` or `~> 1.0`. The version remains unpinned, `mix deps.get` pulls the latest compatible version, and the test exercises a different version than published.

**Why it happens:** Hard-coded version strings in sed patterns don't update automatically.

**How to avoid:** Use the D-04 version-agnostic regex. Optionally add a verification step after sed: `grep -q "== ${VERSION}" examples/hex_consumer/mix.exs` to confirm the substitution occurred.

**Warning signs:** CI passes but the installed dep version differs from `${VERSION}`.

### Pitfall 3: `working-directory` on the rm Step

**What goes wrong:** Adding `working-directory: examples/hex_consumer` to the rm step would require the paths in `run:` to be relative to that directory, making them `lib/hex_consumer_web/oban_powertools_auth.ex` — which works, but is inconsistent with how the step is documented (repo-root relative paths) and slightly confusing.

**Why it happens:** Cargo-culting `working-directory` from adjacent steps.

**How to avoid:** Run the rm from the checkout root (default). Use full repo-root-relative paths in the `run:` command.

### Pitfall 4: sed -i Portability (macOS vs. Linux)

**What goes wrong:** On macOS, `sed -i` requires an explicit empty string argument: `sed -i ''`. On Linux (GitHub Actions ubuntu-latest), `sed -i` takes no argument. The current command already uses `sed -i` without the empty string — correct for ubuntu-latest.

**Why it happens:** BSD sed vs. GNU sed difference.

**How to avoid:** No change needed — CI runs ubuntu-latest. The existing `sed -i` form is correct. Do not add `''`.
