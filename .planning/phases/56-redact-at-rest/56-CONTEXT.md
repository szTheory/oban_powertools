# Phase 56: redact: At-Rest - Context

**Gathered:** 2026-06-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 56 adds compile-time, opt-in **at-rest argument redaction** to
`ObanPowertools.Worker`. A worker declaring `redact: [:ssn, :token]` has those
keys **dropped from `oban_jobs.args`** (key-absent in the JSONB, never written)
at enqueue time â€” strictly **after** the idempotency fingerprint is computed
from the full unredacted args. The dropped field names are recorded in job meta
as `__redacted_fields__` so the row is self-describing, and the native
`/ops/jobs` job detail view surfaces that fact to operators.

This phase delivers **REDACT-01 through REDACT-04 only**. It is at-persist
**deletion**, not encryption, not nulling, not display-time masking.

**Not in this phase:** field-level `encrypt:` (deferred â€” collides with the
idempotency fingerprint, blinds the job filter, leaks via meta/errors);
retroactive scrubbing of already-stored jobs (`Redactor.scrub_past_jobs/2` â€”
needs full Lifeline preview/reason/audit wrapping); redacting from meta,
errors, or stacktraces (those are operational evidence and stay intact);
auto-scrubbing recorded output payloads; a Doctor bypass-detection advisory.

</domain>

<decisions>
## Implementation Decisions

### Redaction intercept point (REDACT-01) â€” SECURITY BOUNDARY
- **D-01:** Redaction lives in a **macro-overridden `new/1,2`** generated by
  `ObanPowertools.Worker.__using__`, following the existing `defoverridable`
  pattern already used for `perform/1`, `timeout`, and the hooks
  (`worker.ex:58,124`). `worker_mod.new/2` is the universal `args â†’ changeset`
  choke-point: overriding it covers **both** `MyWorker.enqueue/2` (â†’
  `Idempotency.transaction/3`) **and** the idiomatic direct
  `MyWorker.new(args) |> Oban.insert()` path. This mirrors how Oban Pro hooks
  the worker changeset build for `encrypted:`, and the Sidekiq client-middleware
  model (runs on every push). A `transaction/3`-only intercept was **rejected**
  because the direct-insert path would silently persist PII.
- **D-02:** Redaction is `Map.drop(args, redacted_fields)` â€” **key-absent**, not
  `nil` and not a `"[REDACTED]"` placeholder. A missing JSONB key is the
  compliance-clean state ("does this column store PII? no").
- **D-03:** **Fingerprint ordering is invariant.** `Idempotency.transaction/3`
  computes `generate_fingerprint(worker_mod, casted_args)` at `idempotency.ex:46`
  from the **full** args, then calls `worker_mod.new(args_map, ...)` at
  `idempotency.ex:81`. Because the fingerprint already runs before `new/2`, the
  override drops fields **after** fingerprinting by construction. Never redact
  in `Args.changeset/2` or `validate/1` â€” `process/1` and the in-memory struct
  must retain values; redaction is an at-persist concern, not a runtime one.
- **D-04:** `__redacted_fields__` meta must be injected **exactly once**. The
  overridden `new/2` injects it; `transaction/3` already calls `new/2`, so do
  NOT also inject it in `transaction/3`. Use the existing deep-merge
  (`idempotency.ex` `deep_merge/2`) so it does not clobber the
  `oban_powertools.idempotency_fingerprint` / limits / deadline meta.

### Cron enqueue coverage (closes a bypass found in research)
- **D-05:** **Fix the cron path.** `cron.ex:422` currently builds jobs with bare
  `Oban.Job.new(args, worker: entry.worker, ...)`, bypassing `worker_mod.new/2`
  entirely â€” so without this fix, cron-scheduled redact-workers would persist
  PII un-redacted. Route cron enqueue through `entry.worker.new(args, ...)` when
  `entry.worker` is an `ObanPowertools.Worker`, so scheduled jobs inherit
  redaction + `__redacted_fields__` meta. Fall back to bare `Oban.Job.new/2`
  for plain `Oban.Worker` cron entries (which have no `redact:` config). This
  is required for genuine universal coverage, not a blessed-path illusion.

### Required-field interaction + path scope (REDACT-01)
- **D-06:** **Auto-exempt redacted fields from `validate_required`.** Today the
  generated `Args.changeset/2` does `validate_required(Keyword.keys(args_config))`
  (`worker.ex:97-101`) â€” **all-or-nothing**, every typed field is required. Since
  `perform/1` re-casts the **stored (redacted)** args on each run
  (`worker.ex:127`), a field that is both typed and redacted would fail
  validation on **every execution** â€” dead-on-arrival without this. A redacted
  field is intentionally absent at runtime, so it is automatically removed from
  the required set at the perform re-cast. No new `required: false` public API is
  added (rejected as redundant ceremony; redact already implies absent-at-runtime).
- **D-07:** **Compile-time typo guard.** Raise `ArgumentError` at `__using__`
  time (beside `validate_args_config!`, `worker.ex:27`) if a `redact:` field is
  **not** a declared `args` schema field. This kills the silent-no-op failure
  mode (a typo'd or wrong-shaped redact key would `Map.drop` nothing and leak).
  Note: this scopes `redact:` to **declared typed fields** in v1.7; that is
  acceptable because it is exactly what makes the static guards possible.
- **D-08:** **Top-level keys only** for v1.7 (`Map.drop`). The typed schema is a
  flat `embedded_schema` (`worker.ex:92-94`) â€” nested paths have nothing to bind
  against and would reintroduce silent-no-op risk. Nested redaction is deferred.
- **D-09:** **Compile-time partition-key guard.** Raise if a `redact:` field is
  also used as a limiter `partition_by: {:args, field}` key â€” otherwise the PII
  (the partition key) lands in meta and defeats redaction (PITFALLS.md).

### Recording integration (the ROADMAP "...and recording integration")
- **D-10:** **Document-only at the recording layer.** `redact:` does NOT touch
  the Phase 55 output-recording pipeline and does NOT mutate `JobRecord.redacted`
  â€” that boolean keeps its honest `false` default (`job_record.ex` `record/3`)
  and `DisplayPolicy.job_recorded/2` (`runtime_config.ex:202`) continues to
  honestly render "not redacted". Setting it `true` would assert a
  payload-cleanliness guarantee the library cannot make (Phase 55 **D-36**:
  display the flag as honest stored metadata, never as proof args were redacted).
  Auto-scrubbing the payload was **rejected** â€” recorded-payload keys are the
  worker's *return-value* keys, a different namespace from arg keys, so
  `Map.drop` over the payload gives false assurance and can silently delete a
  legitimately-named result field (e.g. a freshly-minted `:token`).
- **D-11:** Document explicitly in `guides/workers-and-idempotency.md`:
  *"`redact:` removes fields from args at enqueue; it does NOT scrub recorded
  outputs. Workers must not return redacted/sensitive data from `process/1`."*
- **D-12:** The ROADMAP's "UI annotation" obligation is met at the **args panel**
  (where redaction actually happened) via `meta["__redacted_fields__"]`, fully
  separate from the recorded-output card. This is already the UI-SPEC contract.

### Operator display (REDACT-03, REDACT-04) â€” locked by 56-UI-SPEC.md
- **D-13:** `/ops/jobs` job detail renders `Fields redacted at enqueue:
  [:ssn, :token]` from `meta["__redacted_fields__"]` near the Meta card; no new
  top-level card. Each redacted arg field renders inline in the Args card as
  `field: Redacted at enqueue` so absence reads as intentional.
- **D-14:** `DisplayPolicy.render_job_field/3` (`runtime_config.ex:206`) default
  rendering shows **"Redacted at enqueue"** for any arg field listed in
  `__redacted_fields__`. Copy rules: never use `Hidden`/`Masked`/`Encrypted`
  (the field is absent, not encrypted/display-masked); `[redacted]` remains the
  bounded fallback for a raising/invalid display policy; neutral colors only
  (redaction is expected data minimization, not an error). Read-only â€” no
  reveal/edit action.

### Doctor advisory
- **D-15:** **Defer** the "worker declares `redact:` but meta lacks
  `__redacted_fields__`" bypass-detection advisory. It would require
  worker-module introspection at doctor time, crossing the doctor's deliberate
  DB-only/`pg_catalog` boundary (cf. the Phase 54 expired-deadline check, which
  is pure SQL), and carries a high false-positive surface (no-op drops, jobs
  enqueued before `redact:` was added, retired workers). No REDACT requirement
  asks for it. The cheaper, higher-value safety nets (D-01 universal intercept +
  D-05 cron fix + D-07 compile-time guards) make the supported path
  bypass-resistant instead of auditing for bypass after the fact.

### Atom-vs-string key normalization (cross-cutting â€” for the planner)
- **D-16:** Drop on the **atom-keyed** map at the right boundary. In
  `transaction/3` the args reach `new/2` as `Map.from_struct(casted_struct)` â†’
  **atom keys** (`idempotency.ex:78-81`), so `Map.drop(args_map, [:ssn])` with
  atom keys is correct **there**. Do NOT attempt redaction on post-insert JSONB
  (string keys) or in the `process/1` re-cast path. For direct
  `MyWorker.new(args)` the override must normalize whatever key shape the host
  passes before dropping.
- **D-17:** Store `__redacted_fields__` in meta as **strings**
  (`["ssn", "token"]`, JSONB-native). The UI surfaces the `[:ssn, :token]`
  atom-list presentation form, and `DisplayPolicy.render_job_field/3` matches
  against **string**-keyed JSONB args/meta.

### Claude's Discretion
- Per the user's stated posture ("research deeply, one-shot a coherent
  recommendation set, don't make me think; escalate only new public API / security"),
  all gray areas were researched via parallel subagents grounded in `prompts/`
  and `.planning/research/`, then synthesized into the locked decisions above.
  Two findings were escalated and confirmed by the user: the cron-path bypass
  (D-05 â†’ fix) and the required-field collision (D-06 â†’ auto-exempt + typo guard).
  Downstream agents should implement these decisions, not reopen them.

## Existing Code Insights

### Reusable Assets
- `ObanPowertools.Worker.__using__/1`: already owns Powertools-opt parsing,
  compile-time validation (`validate_args_config!`, `normalize_limits_config!`),
  the typed `embedded_schema`, and the `defoverridable` wrapping pattern. The
  correct and only home for `redact:` parsing, the compile-time guards, the
  `new/1,2` override, and required-set exemption.
- `Idempotency.transaction/3` + `deep_merge/2`: fingerprint-first ordering and
  meta-merge machinery already exist; reuse for `__redacted_fields__` injection.
- `DisplayPolicy.render_job_field/3`: host-owned policy seam with a safe bounded
  fallback already in place â€” extend default rendering for redacted fields.
- `JobRecord` `redacted` field: already present and honestly defaulted `false`;
  reserve for a future feature that actually scrubs payloads.

### Established Patterns
- Compile-time `raise ArgumentError` on contradictory worker opts is the house
  style (`normalize_limits_config!` raises when `:partitioned` lacks
  `partition_by`). The redact typo / required / partition guards follow it exactly.
- `defoverridable` + wrap is how this worker macro already extends Oban behaviour
  (perform, timeout, hooks). Overriding `new` is idiomatic here, not surprising.
- Doctor checks are read-only SQL over `pg_catalog`/`oban_jobs` â€” never module
  introspection. This is why the redact bypass advisory is deferred (D-15).
- Native operator pages use compact bordered cards, honest neutral empty states,
  and text-not-color disclosure â€” the redaction annotation conforms.

### Integration Points
- `worker.ex`: parse/validate `redact:`; generate `__powertools_redact__/0`;
  exempt redacted fields from the required set at re-cast; override `new/1,2`.
- `idempotency.ex`: coordinate single `__redacted_fields__` meta injection;
  preserve fingerprint-before-redact.
- `cron.ex:422`: reroute scheduled enqueue through `entry.worker.new/2`.
- `runtime_config.ex` + `web/jobs_live.ex`: render "Redacted at enqueue" and the
  "Fields redacted at enqueue: [...]" disclosure.
- `guides/workers-and-idempotency.md`: `redact:` support-truth + the
  "not scrubbed from outputs" boundary.

<specifics>
## Specific Ideas

- Keep generated macro code small: put redaction/drop/normalize logic in a small
  internal helper (mirrors `ObanPowertools.Worker.Deadlines`/`Recorder`) rather
  than growing quoted code.
- Make redaction **loud on misconfig**: compile-time raise on typo'd redact keys
  (D-07) and partition-key collisions (D-09); never silently `Map.drop` nothing.
- `__redacted_fields__` stored as sorted string list for stable display/tests.
- The worker's typed args schema must tolerate redacted fields being absent at
  runtime (D-06) â€” tests must prove a typed+redacted worker runs and retries
  cleanly with the field gone from stored args.

</specifics>

<deferred>
## Deferred Ideas

- **`encrypt:` (field-level at-rest encryption)** â€” explicitly deferred in
  PROJECT.md (collides with the args-hashing fingerprint, blinds the job filter,
  leaks via meta/errors/stacktraces). Hosts needing stored-encrypted values
  should encrypt before passing as an arg.
- **Retroactive redaction** (`Redactor.scrub_past_jobs/2`) â€” modifies production
  history and idempotency receipts; belongs in a dedicated module with full
  Lifeline preview/reason/audit wrapping. Out of scope until adopter demand.
- **Nested redaction paths** (`[[:user, :ssn]]`) â€” deferred; typed schema is flat
  in v1.7 and nested paths reintroduce silent-no-op risk.
- **`required: false` per-field arg modifier** â€” not added; auto-exemption (D-06)
  covers the redact case without new public API. Revisit if a non-redact optional
  typed field is ever needed.
- **Doctor redact-bypass advisory** â€” deferred (D-15). **Trigger to build it:** an
  adopter report or support-truth incident where PII is found in `oban_jobs.args`
  for a worker that *did* declare `redact:` (a real bypass with no operator signal).
- **Auto-scrubbing recorded output payloads** â€” rejected as wrong-namespace/false
  assurance (D-10); reserve `JobRecord.redacted = true` for a future feature that
  genuinely scrubs a recorded payload.

None of the above belong in Phase 56.

</deferred>
